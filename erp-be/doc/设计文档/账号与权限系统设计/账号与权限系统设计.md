# 账号与权限系统设计（RBAC + 用户级权限覆盖）

## 1. 背景与目标

**为什么做这个模块**：作为 ERP 系统的基石模块，负责全系统的用户账号管理、角色分配以及细粒度的权限控制。为各业务模块提供坚实且灵活的安全底座。

**解决什么问题**：实现系统的访问控制，防止越权访问；规范前后端权限鉴定标准；解决异步任务无权限上下文的问题。

**非目标是什么**：本期不包含复杂的组织架构树（部门管理和人员管理交由对应系统实现）；不包含复杂的数据权限底层强拦截（避免引发复杂报表的 SQL 解析异常或性能灾难）。

---

## 2. 设计原则

- **认证授权分离**：使用 Spring Security 进行统一的认证与授权拦截。
- **权限收敛到统一模型**：采用 RBAC + 账户直绑权限模型。
- **数据权限业务驱动**：不在底层架构（如 MyBatis 拦截器层）做“一刀切”的强制数据权限拦截，数据可见性管控在具体的 Service 层代码中按需拼装查询条件，保持底层架构绝对轻量。
- **当前态与审计分离**：关联表的复用（Upsert 机制）仅表达“当前态 + 最近变更”，不保证历史完整性，不应作为审计替代。
- **安全优先于便利**：系统采用默认拒绝（deny by default）策略；密码严禁明文存储。

---

## 3. 领域模型设计

### 3.1. Account (账户)

- **继承与关联**：账户禁止物理与逻辑删除，仅允许通过 `enabled = false` 禁用/停用；不支持继承（交由部门/人员系统）；允许关联多个角色。
- **边界授权补丁**：允许账户显式添加 (`effect_type=ALLOW`) 或拒绝 (`effect_type=DENY`) 特定权限作为例外补丁机制。账户级显式授权/拒绝仅允许具备“权限管理”能力的角色执行，并需受授权边界控制。

### 3.2. Role (角色)

- **继承与复用**：角色不允许继承。对于通用权限，应抽取为“基础角色”或“公共角色”（全账户可见但按需绑定），在创建/初始化时自动绑定，允许后续解绑或修改其中权限。允许存在多个基础角色对应不同业务线。
- **授权约束**：角色仅支持权限授予（ALLOW），不支持权限拒绝（DENY）。

### 3.3. Permission (权限资源)

- 权限不仅是鉴权凭证，也是全量资源集合。
- **分类 (Type)**：必须通过 `type` 字段严格区分资源类型：`MENU`、`BUTTON`、`API`、`DATA`、`FILE`、`CAPABILITY`。
- **层级归属**：通过 `businessGroup` 进行业务模块归类。`MENU`/`BUTTON` 采用树形组织；其他类型可挂载到树节点，但授权语义不强依赖树结构。**树结构主要用于 UI 展示与分配，不作为权限判定依据。**
- **唯一性与大小写**：`access_resource` 应全局唯一，不允许重复定义；统一使用小写字母与短横线（-），不使用大写字母。
- **格式规范**：
    - `MENU`：'menu:sys:account'
    - `BUTTON`：'btn:sys:account:add'
    - `API`：系统默认主格式 'domain:resource:action' (例 'sys:account:add')
    - `DATA`：'data:sys:account:show-phone'
    - `FILE`：'file:sys:account:import'
    - `CAPABILITY`：'cap:search:all-deleted'
- **通配符限制**：格式通配符 `*` 仅作为页面查询使用，不作为 `access_resource` 值存储。

### 3.4. Association (角色权限计算规则)

对账户的最终权限判定，按以下顺序执行（匹配即终止）：

1. 若存在账户级显式拒绝规则，则无权限；
2. 否则，若存在账户级显式允许规则，则有权限；
3. 否则，若账户任一启用角色包含该权限，则有权限；
4. 否则，判定为无权限（默认拒绝）。

## 4. 技术架构设计

### 4.1. 登录认证链路与鉴权链路

- **认证拦截**：基于单体应用架构，拦截与校验统一交由 Servlet Filters / Spring Security 拦截链管理。
- **权限判定入口规范**：
    - `MENU`：前端权限。由前端获取并生成Menu菜单，不应通过权限获取菜单配置（如Vue动态路由）。
    - `BUTTON`：前端权限。由前端对应页面获取控制。不代表具备对应 API 权限。
    - `API`：后端权限。由 Spring Security 管理。以 `access_resource` 为唯一标识，与具体 URL/方法解耦，通过注解/配置映射。
    - `DATA`/`FILE`/`CAPABILITY`：通用权限（无前后端区分），不参与统一拦截链路。作为代码级权限，由业务代码按需显式判断。**DATA 权限建议通过权限编码与业务字段映射方式控制，而非直接拼接 SQL 片段。**

### 4.2. Token + Redis 会话模型与缓存失效策略

- **分布式会话**：采用 `JWT/Token + Redis` 架构，支持强制下线和权限刷新。
- **快照缓存**：权限快照应以账户为粒度缓存最终权限集合（合并角色与账户级规则），避免重复计算。
- **失效刷新策略**：权限变更后不**主动刷新 Redis** 中的权限快照，因此不保证当前会话的实时一致性；需通过重新登录或主动刷新触发最新权限集合加载。。

## 5. 数据模型与约束

### 5.1. 表结构与唯一索引

- 核心六表模型：`sys_account`、`sys_role`、`sys_permission` 以及对应的三大中间关联表 `sys_account_role_association`、`sys_role_permission_association`、`sys_account_permission_association`
- `sys_account` 对 `username` 建立唯一索引，确保登录名全局唯一。
- 关联表必须建立联合唯一索引：`(account_id, role_id)`、`(role_id, permission_id)`、`(account_id, permission_id)`。

### 5.2. 逻辑删除与幂等性

- **级联删除**：`Role` 和 `Permission` 允许逻辑删除，删除主实体时**必须同步级联**处理其绑定的关联表记录。
- **绑定与解绑的幂等性约定**：
    - 重复绑定已存在且正常的关联记录，视为成功，不抛异常。
    - 解绑不存在的关联记录，视为成功，不抛异常。

### 5.3. 并发一致性与复用机制 (Upsert)

- **复用机制**：基于双字段唯一索引，关联关系的绑定/解绑不采用新增多条软删除记录的方式，而是通过更新记录的 `enabled` 状态（Upsert）来实现。
- **并发控制**：以数据库唯一约束和事务内幂等更新为主，应用层锁仅作为高竞争热点场景的优化。

## 6. 核心业务流程

### 6.1. 账号/角色/权限基础流程（界面规范）

- **账户管理**：浏览列表、查看详情、角色授权（浏览/添加/绑定/解绑）、停用/启用。
- **角色管理**：浏览列表、查看详情、账户绑定管理、权限绑定管理、新增/修改/删除。
- **资源管理**：资源浏览列表、查看详情、角色绑定管理、新增/修改/删除。

### 6.2. 异步任务系统账号注入

- **痛点**：对于 `@Scheduled`、MQ 消费端及 `@Async`，脱离了 HTTP 生命周期，调用 `SecurityUtils.getCurrentUserId()` 会导致 NPE。
- **系统级虚拟账号兜底机制**：
    1. 系统需预置不可登录的超级保留账号（如 `account_id = -1`, `username = 'SYSTEM_JOB'`）。
    2. 任务入口处（可通过 AOP 切面实现），主动向 `SecurityContext` 注入该“系统账号”认证信息。
    3. **清理闭环**：任务执行结束后（在 `finally` 块中），必须主动调用 `SecurityContextHolder.clearContext()` 清理，防止 ThreadLocal 内存泄漏。

## 7. 安全与审计

### 7.1. 越权分配与登录安全

- **密码存储**：使用 `bcrypt` 强哈希算法（自动加盐）。
- **越权控制**：账户级显式授权仅作为例外补丁机制，需受授权边界严格控制。

### 7.2. 审计字段写入规范

- 依赖 6.2 的虚拟账号注入机制，确保底层业务在写入数据审计字段（`create_by` / `update_by`）时，能正确追溯为“系统定时任务”操作，避免插入脏数据或 NPE。

## 8. 测试策略

### 8.1. Mock SecurityContext 与集成测试约定

- **上下文伪造**：集成/单元测试中，严禁使用真实数据库密码发起 HTTP 登录请求来获取 Token。
- **Mock 机制**：应利用 Spring Security 的 `@WithMockUser`、`@WithUserDetails` 注解，或在 `setUp` 阶段手动向 `SecurityContextHolder` 注入 `UsernamePasswordAuthenticationToken`，低成本模拟会话。
