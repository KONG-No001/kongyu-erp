# 01-账号与权限（RBAC1）系统设计

## 1. 模块概述

作为 ERP 系统的基石模块，负责全系统的用户账号管理、角色分配以及细粒度的权限控制。系统采用RBAC + 账户直绑权限模型，为各业务模块提供坚实且灵活的安全底座。

## 2. 技术框架设计

### 2.1. 认证体系与权限链路设计

- **认证框架**：使用 `Spring Security` 进行统一的认证与授权拦截。
- **密码存储**：使用 Spring Security 默认的 `bcrypt` 强哈希算法（自动加盐），严禁明文存储。
- **会话状态**：采用 `JWT/Token + Redis` 的分布式会话架构，兼顾前后端分离下的无状态跨域调用，以及服务端的集中式会话管理（如强制下线、权限实时刷新）。
- **判定链路**：基于单体应用架构，权限的拦截与校验统一交由 Servlet Filters / Spring Security 拦截链路由与管理。
- **账户权限变更**：用户登录后，其权限快照可缓存于 Redis；在角色或账户权限变更时，不会更改redis，做到账户权限一致性，应通知账户所有人主动刷新页面，使后续请求重新加载最新权限集合。**该策略不保证当前会话内的权限实时一致性，在用户刷新页面或重新发起请求前，仍可能使用旧权限集合。**

**权限快照应以账户为粒度缓存其最终权限集合（已合并角色与账户级规则），避免每次请求重复计算。**

### 2.2. 数据权限模型

- **架构原则**：不在底层架构（如 MyBatis 拦截器层）做“一刀切”的强制数据权限拦截，避免引发复杂报表的 SQL 解析异常或性能灾难。
- **实现方式**：数据可见性管控遵循业务驱动原则，在具体的 `Service` 层代码中按需拼装查询条件，保持底层架构绝对轻量。

### 2.3. 权限资源定义 (Permission 实体约定)

`Permission` 表不仅是后端接口鉴权凭证，也是全量资源集合。

- **分类 (Type)**：必须通过 `type` 字段严格区分资源类型：`MENU`（菜单）、`BUTTON`（按钮）、`API`（接口）、`DATA`（数据）、`FILE`（文件）、`GENERIC`(权限)。
- **层级归属**：通过 `businessGroup` 字段对资源进行业务模块归类。菜单、按钮等展示型资源采用树形组织；`API`、`DATA`、`FILE` 等资源可按业务分组挂接到`parent_id`树节点，但其授权语义不强依赖树结构本身。权限标识为`access_resource`

**树结构主要用于 UI 展示与分配，不作为权限判定依据**

### 2.5. 角色定义 (Role 实体约定)

- **角色是否允许继承**：不能，对于那些会被所有角色关联的权限，应当抽取成为“基础角色”或“公共角色”，并在账户创建或初始化时自动勾选。

- **基础角色是否允许解绑**：可以，需要在权限管理里解绑。

- **是否允许修改基础角色的权限**：可以，如果其中的权限已经不适用了，可以解除绑定或者添加权限。

- **是否存在多个基础角色**：允许，不同业务应该有不同的基础角色

- **基础角色是全账户统一，还是按业务线不同？**：基础角色全账户都可见，但是按需绑定

**本系统不采用角色继承，以降低权限推导复杂度与维护成本；对公共能力通过基础角色复用，对个别差异通过账户级显式授权补充。**

**角色仅支持权限授予（ALLOW），不支持权限拒绝（DENY）。**

### 2.6. 账户 (Account 实体约定)

- **账户是否允许继承**：不能，该业务逻辑应该交由部门管理和人员管理系统实现
- **是否允许账户关联多个角色**：可以。
- **是否允许账户显式添加某个权限**：可以，对于不满足账户现有的权限，而额外新增角色开销过大，可显式添加权限(`effect_type=ALLOW`)关系
- **是否允许账户显式拒绝某个权限**：可以，对于该账户所有的权限超过了应有的权限，可显式添加(`effect_type=DENY`)权限关系

**账户级显式授权仅作为例外补丁机制使用，需受授权边界控制；账户级显式授权/拒绝仅允许具备“权限管理”能力的角色执行，并需受授权边界控制。**

### 2.7. 权限计算规则

对账户的最终权限判定，按以下顺序执行：

1. 若存在账户级显式拒绝规则，则判定为无权限；
2. 否则，若存在账户级显式允许规则，则判定为有权限；
3. 否则，若账户任一启用角色包含该权限，则判定为有权限；
4. 否则，判定为无权限。

**系统采用默认拒绝（deny by default）策略。**

### 2.8. 权限判定入口

对权限的判断入口，应按一下约定

- MENU：前端权限；应该由前端获取并生成Menu菜单，不应该通过权限获取菜单配置，比如VUE动态路由
- BUTTON：前端权限；应该由前端对应页面获取。
- API：后端权限；该权限由后端 Spring Security 拦截链路由与管理。**API 权限应以 access_resource 为唯一标识，与具体 URL/方法解耦，通过注解或配置进行映射。**
- DATA：通用权限，无前后端区分；该权限作为页面查询、数据详情等特定数据的展示是否，例如表格展示特定订单字段，详情展示特定价格字段。作为代码级的权限，由业务和开发自行决定是否需要。**DATA 权限建议通过权限编码与业务字段映射方式控制，而非直接拼接 SQL 片段。权限缓存应在用户重新登录或主动刷新页面时重新加载，避免长期使用过期权限。**
- FILE：通用权限，无前后端区分；该权限作为特定文件生成、导出、下载、上传所用。其适用范围比DATA更小，通常作为某些数据最后的保险，例如财会数据的导出和上传。作为代码级的权限，由业务和开发自行决定是否需要。
- GENERIC：通用权限；无前后端区分；非特定类型的权限，比如可使用某个查询字段。作为代码级的权限，由业务和开发自行决定是否需要。

### 2.8. 权限access_resource格式

对权限不同类型的格式

- MENU：'menu:sys:account'
- BUTTON：'btn:sys:account:add'、'btn:sys:account:edit'
- API： 'sys:account:add'、'sys:account:edit'、'sys:account:disable'
- DATA：'data:sys:account:show-real-name','data:sys:account:show-phone'
- FILE：'file:sys:account:import','file:sys:account:export'
- GENERIC：'cap:search:all-deleted'、'cap:report:cross-department-view'

**API 权限采用系统默认主格式 domain:resource:action；其他非接口权限通过显式类型前缀进行区分。**
**DATA、FILE、CAP 类型权限不参与统一拦截链路，由业务代码按需显式判断。**
**BUTTON 权限仅用于前端控制，不代表具备对应 API 权限；API 权限需独立校验。**

### 2.8. 查询权限access_resource格式通配符*

格式通配符*仅作为页面查询使用，不作为 access_resource 值存储

### 2.9. access_resource 唯一性

access_resource 应全局唯一，不允许重复定义

### 2.10. access_resource 大小写规范

access_resource 统一使用小写字母与短横线（-），不使用大写字母

## 4. 数据库模型与约束

### 4.1. 核心模型

六表模型：`Account`、`Role`、`Permission`，以及中间关联表 `RoleAccountAssociation`、`PermissionRoleAssociation`、`PermissionAccountAssociation`。

### 4.2. 数据库约束

- `sys_role_account_association`：建立联合唯一索引 `(role_id, account_id)`。
- `sys_permission_role_association`：建立联合唯一索引 `(role_id, permission_id)`。
- `sys_permission_account_association`：建立联合唯一索引 `(account_id, permission_id)`。
- `sys_account`：对 `username` 建立唯一索引，确保登录名全局唯一。

### 4.3. 软删除与停用机制

- **账户 (Account)**：绝对禁止物理与逻辑删除，仅允许通过 `enabled = false` 进行禁用/停用，以保证历史业务单据的关联一致性。
- **角色 (Role) 与权限 (Permission)**：允许逻辑删除。当主实体被逻辑删除时，**必须同步级联**处理其绑定的关联表记录。
- **关联表的复用 (Upsert 机制)**：基于 4.2 的双字段唯一索引，关联关系的绑定/解绑不采用新增多条软删除记录的方式，而是通过更新记录的 `enabled`（启用/停用）状态来实现。这既保留了操作痕迹，又完美避开了唯一键冲突。**当前表仅表达“当前态 + 最近变更”，不保证历史完整性，不应作为审计替代。**

### 4.4. 绑定与解绑的幂等性约定

- **重复绑定**：绑定已存在且正常的关联记录，视为成功，不抛出异常。
- **空解绑**：解绑不存在的关联记录，视为成功，不抛出异常。

### 4.5. 并发控制要求

- 并发一致性以数据库唯一约束和事务内幂等更新为主，应用层锁仅作为高竞争热点场景下的性能优化手段，而非正确性的唯一保障。

## 5. 开发与测试上下文约定

### 5.1. 测试环境权限注入（单元/集成测试）

- **上下文伪造**：在进行后端集成测试或单元测试时，严禁使用真实数据库密码发起 HTTP 登录请求来获取 Token。
- **Mock 机制**：应利用 Spring Security 提供的 `@WithMockUser`、`@WithUserDetails` 注解，或在测试的 `setUp` 阶段手动向 `SecurityContextHolder` 注入 `UsernamePasswordAuthenticationToken`，以极低成本模拟所需的用户会话与权限集合。

### 5.2. 定时任务与异步线程的“系统账号”约定

- **无请求上下文痛点**：对于 `@Scheduled` 定时任务、MQ 消费端以及 `@Async` 异步线程，由于脱离了 HTTP 请求生命周期，原有的 Token 解析与拦截器将失效。此时若底层业务（如创建订单、更新状态）调用类似 `SecurityUtils.getCurrentUserId()` 的方法，会导致抛出异常或记录为空。
- **系统级虚拟账号兜底**：系统需预置一个不可登录的超级保留账号（例如：`account_id = -1`，`username = 'SYSTEM_JOB'`）。
- **执行上下文注入**：
    1. 在定时任务或消费端的入口处（可通过 AOP 切面实现），主动向 `SecurityContext` 注入该“系统账号”的认证信息。
    2. 确保底层业务在写入数据审计字段（`create_by` / `update_by`）时，能够正确追溯为“系统定时任务”操作，避免插入脏数据或引发 NPE。
    3. **清理闭环**：任务执行结束后（在 `finally` 块中），必须主动调用 `SecurityContextHolder.clearContext()` 进行清理，防止线程池复用导致的上下文污染（ThreadLocal 内存泄漏）。
