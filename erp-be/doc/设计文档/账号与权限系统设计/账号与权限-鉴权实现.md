* Spring Security 接入方式

本文档属于**详细设计 / 开发文档**，直接服务于后端开发实现。

## 3. 总体实现目标

鉴权实现必须满足以下目标：

1. 登录后可基于 Token 恢复当前登录用户身份。
2. 后端 API 鉴权统一基于 `access_resource` 进行判定，与 URL/方法解耦。
3. 最终权限集合需按账户粒度缓存，减少重复查询与计算。
4. 权限计算必须遵循既定顺序：账户拒绝 → 账户允许 → 角色允许 → 默认拒绝。
5. `DATA / FILE / CAPABILITY` 权限不进入统一拦截链路，由业务代码显式调用判定方法。
6. 异步任务和定时任务场景下，必须具备系统账号注入与上下文清理机制。

---

## 4. 总体架构

## 4.1 组件划分

建议后端实现拆分为以下组件：

* `LoginController`：登录接口
* `AuthService`：认证服务
* `TokenService`：Token 生成、解析、失效处理
* `CurrentUserService`：当前登录用户上下文获取
* `PermissionResolveService`：最终权限集合解析
* `PermissionCacheService`：权限快照缓存
* `PermissionEvaluatorImpl`：Spring Security 权限判定实现
* `AccessResourceMappingRegistry`：接口到 `access_resource` 的映射注册中心
* `SystemAccountContextService`：异步线程系统账号注入
* `SecurityUserDetails`：登录态用户对象
* `JwtAuthenticationFilter`：Token 解析过滤器

---

## 4.2 数据流总览

### 登录阶段

1. 用户提交用户名密码
2. 校验账号状态与密码
3. 生成 Token
4. 计算并缓存账户权限快照
5. 返回 Token 给前端

### 请求阶段

1. 请求携带 Token 进入过滤器
2. 解析 Token，获取账户 ID
3. 校验 Token 和会话状态
4. 从 Redis 读取权限快照
5. 构造 Spring Security 上下文
6. 接口进入 `@PreAuthorize("hasPermission(...)")`
7. 由自定义 `PermissionEvaluator` 判定权限

## 5. 登录认证实现设计

## 5.1 登录接口

### 接口建议

`POST /api/auth/login`

### 入参

```json
{
  "username": "admin",
  "password": "123456"
}
```

### 出参

```json
{
  "token": "xxxxx",
  "accountId": 1,
  "username": "admin"
}
```

---

## 5.2 登录流程

### 步骤 1：查询账户

按 `username` 查询 `sys_account`，校验：

* 是否存在
* 是否启用 `enabled = 1`

### 步骤 2：校验密码

使用 `PasswordEncoder.matches(rawPassword, encodedPassword)` 进行 bcrypt 校验。系统设计要求密码必须使用 bcrypt 强哈希存储。

### 步骤 3：构造登录主体

构造 `SecurityUserDetails`，至少包含：

* `accountId`
* `username`
* `enabled`
* `isSystemAccount`
* `permissionSnapshot`（可选，不建议直接塞满）

### 步骤 4：生成 Token

Token 中建议最少放入：

* `accountId`
* `username`
* `issuedAt`
* `expireAt`

不建议把完整权限集合塞进 JWT 本体，避免权限变更后旧 Token 携带旧权限。

### 步骤 5：建立 Redis 会话

Redis 中建议维护：

* Token 与账户映射
* 账户级权限快照
* 可选的会话状态信息

### 步骤 6：加载权限快照

登录成功后立即调用 `PermissionResolveService.resolvePermissions(accountId)` 计算最终权限集合，并写入 Redis。系统设计要求权限快照以账户为粒度缓存最终权限集合。

## 6. Token 与 Redis 会话设计

## 6.1 Redis Key 建议

建议采用如下 Key 结构：

```text
auth:token:{token}
auth:permission:{accountId}
auth:session:{accountId}
```

### 示例

```text
auth:token:eyJhbGciOi...
auth:permission:1001
auth:session:1001
```

---

## 6.2 Value 建议

### `auth:token:{token}`

存储：

```json
{
  "accountId": 1001,
  "username": "admin",
  "expireAt": 1777000000000
}
```

### `auth:permission:{accountId}`

存储：

```json
{
  "accountId": 1001,
  "permissions": [
    "sys:account:add",
    "sys:account:edit",
    "btn:sys:account:add",
    "menu:sys:account",
    "data:sys:account:show-phone"
  ],
  "loadedAt": 1776322974114
}
```

### `auth:session:{accountId}`

可选存储：

```json
{
  "accountId": 1001,
  "status": "ACTIVE"
}
```

## 6.3 会话失效策略

本期策略按系统设计执行：

* 权限变更后**不主动刷新 Redis 中的权限快照**
* 也不追求当前会话的实时一致性
* 需通过重新登录或主动刷新触发最新权限集合加载。

因此：

* Token 仍可校验成功
* 但权限快照可能短暂滞后
* 该行为属于设计允许范围

## 7. 请求鉴权实现设计

## 7.1 过滤器位置

建议在 Spring Security 过滤链中加入：

* `JwtAuthenticationFilter`

位置建议：

* 在 `UsernamePasswordAuthenticationFilter` 之前执行

---

## 7.2 过滤器职责

`JwtAuthenticationFilter` 负责：

1. 从请求头读取 `Authorization: Bearer {token}`
2. 解析 Token
3. 校验 Token 是否存在、是否过期
4. 查询 Redis 中的 token/session 信息
5. 加载账户权限快照
6. 构造 `Authentication` 放入 `SecurityContextHolder`

---

## 7.3 过滤器伪代码

```java
public void doFilterInternal(...) {
    String token = resolveToken(request);
    if (token == null) {
        filterChain.doFilter(request, response);
        return;
    }

    TokenPayload payload = tokenService.parse(token);
    if (payload == null || tokenService.isExpired(payload)) {
        filterChain.doFilter(request, response);
        return;
    }

    SessionInfo sessionInfo = tokenService.loadSession(token);
    if (sessionInfo == null) {
        filterChain.doFilter(request, response);
        return;
    }

    Set<String> permissions = permissionCacheService.getPermissions(payload.getAccountId());
    SecurityUserDetails userDetails = buildUserDetails(payload, permissions);

    Authentication authentication =
        new UsernamePasswordAuthenticationToken(userDetails, null, Collections.emptyList());

    SecurityContextHolder.getContext().setAuthentication(authentication);
    filterChain.doFilter(request, response);
}
```

## 8. 权限快照加载与计算实现

## 8.1 目标

将数据库中的角色权限、账户直绑权限合并成最终权限集合 `Set<String>`。

---

## 8.2 输入

输入参数：

* `accountId`

---

## 8.3 输出

输出结果：

* 最终权限集合 `Set<String> permissions`

其中元素为 `access_resource`，例如：

* `sys:account:add`
* `btn:sys:account:add`
* `data:sys:account:show-phone`

---

## 8.4 数据来源

需要查询以下数据：

1. 启用中的账户记录 `sys_account`
2. 启用中的账户角色绑定 `sys_account_role_association`
3. 启用且未删除的角色 `sys_role`
4. 启用中的角色权限绑定 `sys_role_permission_association`
5. 启用且未删除的权限 `sys_permission`
6. 启用中的账户权限绑定 `sys_account_permission_association`

这些约束与前置数据库详细设计保持一致。

---

## 8.5 计算规则

### 步骤 1：加载角色权限集合

查询账户所有启用角色，再查询角色启用中的权限，得到：

```java
Set<String> rolePermissions
```

### 步骤 2：加载账户权限规则

查询账户直绑权限，拆成：

```java
Set<String> userAllowPermissions
Set<String> userDenyPermissions
```

### 步骤 3：合并规则

最终结果为：

```java
finalPermissions = (rolePermissions ∪ userAllowPermissions) - userDenyPermissions
```

该结果与系统设计中的判定顺序等价。

---

## 8.6 推荐伪代码

```java
public Set<String> resolvePermissions(Long accountId) {
    Set<String> rolePermissions = permissionRepository.findRolePermissionsByAccountId(accountId);
    Set<String> userAllowPermissions = permissionRepository.findAccountPermissions(accountId, "ALLOW");
    Set<String> userDenyPermissions = permissionRepository.findAccountPermissions(accountId, "DENY");

    Set<String> result = new HashSet<>(rolePermissions);
    result.addAll(userAllowPermissions);
    result.removeAll(userDenyPermissions);

    return result;
}
```

## 9. Spring Security 权限判定实现

## 9.1 推荐实现方式

建议使用：

* `@PreAuthorize("hasPermission('sys:account:add')")`

由自定义 `PermissionEvaluator` 处理。

系统设计已明确 API 权限以 `access_resource` 为唯一标识，与 URL/方法解耦，通过注解或配置映射。

---

## 9.2 自定义 PermissionEvaluator

### 接口实现

实现 `org.springframework.security.access.PermissionEvaluator`

### 核心逻辑

* 从 `Authentication` 中取出 `SecurityUserDetails`
* 读取其权限集合
* 判断目标 `access_resource` 是否存在

---

## 9.3 示例代码

```java
@Component
public class PermissionEvaluatorImpl implements PermissionEvaluator {

    @Override
    public boolean hasPermission(Authentication authentication, Object targetDomainObject, Object permission) {
        if (authentication == null || permission == null) {
            return false;
        }

        Object principal = authentication.getPrincipal();
        if (!(principal instanceof SecurityUserDetails)) {
            return false;
        }

        SecurityUserDetails user = (SecurityUserDetails) principal;
        String accessResource = String.valueOf(permission);

        return user.getPermissions().contains(accessResource);
    }

    @Override
    public boolean hasPermission(Authentication authentication, Serializable targetId, String targetType, Object permission) {
        return hasPermission(authentication, null, permission);
    }
}
```

---

## 9.4 使用示例

```java
@PreAuthorize("hasPermission('sys:account:add')")
@PostMapping("/api/accounts")
public Result<Void> createAccount(@RequestBody AccountCreateRequest request) {
    ...
}
```

## 10. API 与 access_resource 的映射设计

## 10.1 原则

* API 权限不直接使用 URL 字符串进行判定
* URL 与方法只作为路由定位
* 真正权限点使用 `access_resource`

这样可避免路由调整导致权限码大规模变化。

---

## 10.2 映射方式

推荐两种方式，二选一即可：

### 方式 A：注解硬编码（推荐）

直接在 Controller/Service 方法上写：

```java
@PreAuthorize("hasPermission('sys:account:edit')")
```

优点：

* 直观
* 修改简单
* 无需额外注册中心

### 方式 B：配置映射

在配置文件中维护：

```yaml
security:
  api-permissions:
    - path: /api/accounts
      method: POST
      access-resource: sys:account:add
```

优点：

* 统一管理
  缺点：
* 成本更高
* 维护复杂

本期建议优先使用**方式 A**。

## 11. 业务代码显式权限判定实现

系统设计要求 `DATA / FILE / CAPABILITY` 不参与统一拦截链路，由业务代码按需显式判断。

---

## 11.1 提供统一工具类

建议提供：

* `PermissionService.hasPermission(String accessResource)`
* `SecurityUtils.hasPermission(String accessResource)`

---

## 11.2 示例

### DATA 权限

```java
if (permissionService.hasPermission("data:sys:account:show-phone")) {
    vo.setPhone(entity.getPhone());
}
```

### FILE 权限

```java
if (!permissionService.hasPermission("file:sys:account:export")) {
    throw new ForbiddenException("无导出权限");
}
```

### CAPABILITY 权限

```java
if (permissionService.hasPermission("cap:search:all-deleted")) {
    query.setIncludeDeleted(true);
}
```

---

## 11.3 推荐实现

```java
@Service
public class PermissionService {

    public boolean hasPermission(String accessResource) {
        Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
        if (authentication == null) {
            return false;
        }

        Object principal = authentication.getPrincipal();
        if (!(principal instanceof SecurityUserDetails)) {
            return false;
        }

        return ((SecurityUserDetails) principal).getPermissions().contains(accessResource);
    }
}
```

## 13. 异步任务系统账号注入实现

系统设计已要求对 `@Scheduled`、MQ、`@Async` 等场景注入系统账号，并在 finally 中清理上下文。

---

## 13.1 系统账号建议

预置：

* `accountId = -1`
* `username = SYSTEM_JOB`

权限集合建议：

* 可为空集合
* 或按需加载系统内置权限

如果系统任务需要无条件执行管理动作，建议在业务中识别 `isSystemAccount = true`。

---

## 13.2 注入实现方式

建议封装统一方法：

```java
public void runAsSystem(Runnable runnable)
```

---

## 13.3 示例实现

```java
public void runAsSystem(Runnable runnable) {
    try {
        SecurityUserDetails systemUser = new SecurityUserDetails();
        systemUser.setAccountId(-1L);
        systemUser.setUsername("SYSTEM_JOB");
        systemUser.setSystemAccount(true);
        systemUser.setPermissions(Collections.emptySet());

        Authentication authentication =
            new UsernamePasswordAuthenticationToken(systemUser, null, Collections.emptyList());

        SecurityContextHolder.getContext().setAuthentication(authentication);

        runnable.run();
    } finally {
        SecurityContextHolder.clearContext();
    }
}
```

## 14. 异常处理设计

## 14.1 认证异常

当未登录、Token 无效、会话不存在时：

* 返回 `401`

## 14.2 鉴权异常

当权限不足时：

* 返回 `403`

## 14.3 业务异常

当业务代码主动判定 `DATA / FILE / CAPABILITY` 权限失败时：

* 建议统一抛出 `ForbiddenException`
* 由全局异常处理器转换为统一返回结构

---

## 15. 缓存重载策略

## 15.1 登录时重载

登录成功后，立即重算并写入权限快照。

## 15.2 主动刷新时重载

前端刷新页面或重新获取会话信息时，允许触发权限快照重算。

## 15.3 后台不主动推送

本期不实现：

* 权限变更后实时广播
* Redis 主动失效推送
* WebSocket 通知前端刷新

这些属于后续优化内容，与系统设计一致。
