# 1. 文档目的

本文档用于定义系统统一的接口设计规范，包括：

* API 路径规范
* 请求方式规范
* 返回结构规范
* 分页规范
* 权限标识规范
* 错误码规范（基础约定）

用于指导：

* 后端接口开发
* 前后端联调
* 接口测试与 Mock
* 多模块统一风格

---

# 2. 适用范围

本规范适用于：

* ERP 系统所有后端接口
* 所有业务模块（账号、订单、库存、财务等）

所有模块接口设计必须遵循本规范。

---

# 3. 总体设计原则

* 统一性：所有接口遵循统一规范
* 可扩展性：支持后续模块扩展与版本演进
* 清晰性：接口语义明确，避免歧义
* 解耦性：接口路径不依赖数据库结构
* 安全性：接口必须接入统一鉴权机制

---

# 4. API 路径规范

---

## 4.1 基本结构

所有接口统一采用：

```text
/api/{domain}/{resource}
```

### 示例

```text
/api/sys/accounts
/api/sys/roles
/api/order/orders
/api/finance/statements
```

## 4.2 命名规则

| 项目           | 规则                                  |
| ------------ | ----------------------------------- |
| domain（业务组）  | 小写英文，表示业务域（如 sys / order / finance） |
| resource（资源） | 小写复数名词（accounts / roles / orders）   |
| 分隔符          | 使用 `/`                              |
| 大小写          | 全小写                                 |
| 单词连接         | 使用 `-`（如 cross-department）          |

## 4.3 RESTful 约定

| 操作   | 方法     | 路径                            |
| ---- | ------ | ----------------------------- |
| 查询列表 | GET    | /api/{domain}/{resource}      |
| 查询详情 | GET    | /api/{domain}/{resource}/{id} |
| 新增   | POST   | /api/{domain}/{resource}      |
| 修改   | PUT    | /api/{domain}/{resource}/{id} |
| 删除   | DELETE | /api/{domain}/{resource}/{id} |

## 4.4 Action 扩展规则

在以下场景允许使用 action 后缀：

| 场景   | 示例           |
| ---- | ------------ |
| 分页查询 | /page        |
| 状态变更 | /{id}/status |
| 批量操作 | /batch/*     |

### 示例

```text
POST   /api/sys/accounts/page
PATCH  /api/sys/accounts/{id}/status
PATCH  /api/sys/accounts/status/batch
```

## 4.5 禁止事项

* 禁止使用动词作为资源路径（如 `/getAccounts` ❌）
* 禁止混用大小写
* 禁止使用拼音
* 禁止使用单数资源名（如 `/account` ❌）

# 5. 请求规范

## 5.1 请求头

```http
Authorization: Bearer {token}
Content-Type: application/json
```

## 5.2 请求体

* 所有复杂查询必须使用 JSON Body
* 不推荐使用 URL 拼接复杂参数


# 6. 返回结构规范

---

## 6.1 通用返回结构

所有接口统一返回：

```json
{
  "code": 20000,
  "message": "success",
  "data": {},
  "timestamp": 1776322974114,
  "traceId": "abc123"
}
```

---

## 6.2 字段说明

| 字段        | 说明               |
| --------- | ---------------- |
| code      | 业务状态码，20000 表示成功 |
| message   | 响应消息             |
| data      | 返回数据             |
| timestamp | 服务端响应时间          |
| traceId   | 请求链路追踪 ID        |

---

## 6.3 返回规则

* 成功：`code = 20000`
* 失败：返回对应错误码
* 不返回 HTTP 200 以外状态（统一由 code 控制）

---

# 7. 分页规范

---

## 7.1 分页接口规则

* 统一使用 **POST**
* 路径统一为 `/page`
* 请求参数使用 `PageRequest`
* 返回结构为 `Result<PageResult<T>>`

---

## 7.2 示例

### 请求

```json
{
  "page": 1,
  "size": 10,
  "condition": {}
}
```

---

### 返回

```json
{
  "code": 20000,
  "data": {
    "total": 100,
    "list": []
  }
}
```

# 8. 权限规范

---

## 8.1 权限标识（access_resource）

统一采用：

```text
domain:resource:action
```

### 示例

```text
sys:account:add
sys:account:edit
order:order:view
```

---

## 8.2 权限控制方式

* API 权限：通过 Spring Security 拦截
* DATA / FILE / CAP：业务代码显式判断

---

# 9. 错误码规范（基础）

---

## 9.1 成功码

```text
20000 = 成功
```

---

## 9.2 基础错误码

| code  | 说明   |
| ----- | ---- |
| 40000 | 参数错误 |
| 40100 | 未登录  |
| 40300 | 无权限  |
| 50000 | 系统异常 |

---

# 10. 版本策略（基础约定）

---

## 10.1 当前策略

* 默认不在 URL 中体现版本号
* 通过兼容策略保证接口稳定

---

## 10.2 升级策略

当接口发生重大变更时：

* 新接口新增，不覆盖旧接口
* 或通过 `/api/v1/` 引入版本

---

# 11. 统一约定总结

---

所有接口必须遵循：

* `/api/{domain}/{resource}` 结构
* RESTful 方法语义
* Result 统一返回结构
* 分页统一 `/page`
* 权限统一 `access_resource`
* 鉴权统一 Spring Security
