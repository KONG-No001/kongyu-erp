# 01-账号与权限（RBAC1）系统设计

## 1. 模块概述

作为 ERP 系统的基石模块，负责全系统的用户账号管理、角色分配以及细粒度的权限控制。系统采用标准的 RBAC1（角色分层）模型，支持角色与权限的父子级继承，为各业务模块提供坚实且灵活的安全底座。

## 2. 技术框架设计

### 2.1. 认证体系与权限链路设计

- **认证框架**：使用 `Spring Security` 进行统一的认证与授权拦截。
- **密码存储**：使用 Spring Security 默认的 `bcrypt` 强哈希算法（自动加盐），严禁明文存储。
- **会话状态**：采用 `JWT/Token + Redis` 的分布式会话架构，兼顾前后端分离下的无状态跨域调用，以及服务端的集中式会话管理（如强制下线、权限实时刷新）。
- **判定链路**：基于单体应用架构，权限的拦截与校验统一交由 Servlet Filters / Spring Security 拦截链路由与管理。

### 2.2. 数据权限模型

- **架构原则**：不在底层架构（如 MyBatis 拦截器层）做“一刀切”的强制数据权限拦截，避免引发复杂报表的 SQL 解析异常或性能灾难。
- **实现方式**：数据可见性管控遵循业务驱动原则，在具体的 `Service` 层代码中按需拼装查询条件，保持底层架构绝对轻量。

### 2.3. 权限资源定义 (Permission 实体约定)

`Permission` 表不仅是后端接口鉴权凭证，也是全量资源集合。

- **分类 (Type)**：必须通过 `type` 字段严格区分资源类型：`MENU`（菜单）、`BUTTON`（按钮）、`API`（接口）、`DATA`（数据）、`FILE`（文件）。
- **层级归属**：通过 `businessGroup` 字段对资源进行业务模块归类，同时依赖 `parent_id` 形成严格的树形结构，辅助前端构建级联的权限分配树。

### 2.4. 角色定义 (Role 实体约定)

- **角色层级**：通过 `parent_id` 维护角色的父子级拓扑。上级角色可自动继承或管控下级角色的权限，完整落实 RBAC1 的分层特性。

## 4. 数据库模型与约束

### 4.1. 核心模型

五表标准模型：`Account`、`Role`、`Permission`，以及中间关联表 `RoleAccountAssociation`、`PermissionRoleAssociation`。

### 4.2. 数据库约束

- `sys_role_account_association`：建立联合唯一索引 `(role_id, account_id)`。
- `sys_permission_role_association`：建立联合唯一索引 `(role_id, permission_id)`。
- `sys_account`：对 `username` 建立唯一索引，确保登录名全局唯一。

### 4.3. 软删除与停用机制

- **账户 (Account)**：绝对禁止物理与逻辑删除，仅允许通过 `enabled = false` 进行禁用/停用，以保证历史业务单据的关联一致性。
- **角色 (Role) 与权限 (Permission)**：允许逻辑删除。当主实体被逻辑删除时，**必须同步级联**处理其绑定的关联表记录。
- **关联表的复用 (Upsert 机制)**：基于 4.2 的双字段唯一索引，关联关系的绑定/解绑不采用新增多条软删除记录的方式，而是通过更新记录的 `enabled`（启用/停用）状态来实现。这既保留了操作痕迹，又完美避开了唯一键冲突。

### 4.4. 绑定与解绑的幂等性约定

- **重复绑定**：绑定已存在且正常的关联记录，视为成功，不抛出异常。
- **空解绑**：解绑不存在的关联记录，视为成功，不抛出异常。

### 4.5. 并发控制要求

- 应用层需使用并发锁（如基于关联 ID 的细粒度内存锁或 Redis 分布式锁）进行防重校验，防止高频重复请求击穿数据库。

## 5. 开发与测试上下文约定

### 5.1. 测试环境权限注入（单元/集成测试）

- **上下文伪造**：在进行后端集成测试或单元测试时，严禁使用真实数据库密码发起 HTTP 登录请求来获取 Token。
- **Mock 机制**：应利用 Spring Security 提供的 `@WithMockUser`、`@WithUserDetails` 注解，或在测试的 `setUp` 阶段手动向 `SecurityContextHolder` 注入 `UsernamePasswordAuthenticationToken`，以极低成本模拟所需的用户会话与权限集合。

### 5.2. 定时任务与异步线程的“系统账号”约定

- **无请求上下文痛点**：对于 `@Scheduled` 定时任务、MQ 消费端以及 `@Async` 异步线程，由于脱离了 HTTP 请求生命周期，原有的 Token 解析与拦截器将失效。此时若底层业务（如创建订单、更新状态）调用类似 `SecurityUtils.getCurrentUserId()` 的方法，会导致抛出异常或记录为空。
- **系统级虚拟账号兜底**：系统需预置一个不可登录的超级保留账号（例如：`account_id = -1`，`username = 'SYSTEM_JOB'`）。
- **执行上下文注入**：
  1. 在定时任务或消费端的入口处（可通过 AOP 切面实现），主动向 `SecurityContext` 注入该“系统账号”的认证信息。
  2. 确保底层业务在写入数据审计字段（`create_by` / `update_by`）时，能够正确追溯为“系统定时任务”操作，避免插入脏数据或引发 NPE。
  3. **清理闭环**：任务执行结束后（在 `finally` 块中），必须主动调用 `SecurityContextHolder.clearContext()` 进行清理，防止线程池复用导致的上下文污染（ThreadLocal 内存泄漏）。
