# 01-账号与权限（RBAC）系统设计

## 1. 模块概述
作为 ERP 系统的基石模块，负责全系统的用户账号管理、角色分配以及细粒度的权限控制。系统采用标准的 RBAC（基于角色的访问控制）模型，为各业务模块提供安全底座。

## 2. 领域模型与数据库约束
- **核心模型**：五表标准模型（Account, Role, Permission, RoleAccountAssociation, PermissionRoleAssociation）。
- **数据库级物理防线**：
  - `sys_role_account_association` 需添加联合唯一索引 `(role_id, account_id, deleted)`。
  - `sys_permission_role_association` 需添加联合唯一索引 `(role_id, permission_id, deleted)`。
  - `sys_account` 需对 `username` 建立唯一索引。

## 3. 核心业务规则与边界
### 3.1 软删除与停用级联规则
- **账户 (Account)**：绝对禁止物理与逻辑删除，只允许通过 `enabled = false` 进行禁用/停用。
- **角色 (Role) 与权限 (Permission)**：允许逻辑删除。当主实体被删除时，**必须同步级联删除**与其绑定的所有关联表记录（例如：删除 Role1 时，必须同步将 Role1 对应的 `role_account` 和 `permission_role` 关联记录设为逻辑删除）。
- **绑定恢复机制**：对已逻辑删除的关联关系，重新绑定时直接更新为 `deleted=0`，同步重置 `enabled=1`，并刷新审计时间。

### 3.2 绑定与解绑的幂等性约定
- **重复绑定**：绑定已存在且正常的关联记录，视为成功，不抛异常。
- **空解绑**：解绑不存在的关联记录，视为成功，不抛异常。

### 3.3 并发控制要求
- 应用层需使用并发锁（基于关联 ID 的细粒度内存锁等）防止高频重复请求击穿数据库。

## 4. 认证体系与权限链路设计
- **认证框架**：采用 Spring Boot Security 进行自定义认证与授权拦截。
- **密码存储**：使用 Spring Security 默认的 `bcrypt` 强哈希算法（自动加盐），严禁明文存储。
- **会话状态**：采用基于 Session 的状态维持（支撑后续对接外部 API 调用的需求）。
- **判定链路**：基于单体应用架构，权限的拦截与校验统一交由 Servlet Filters / Spring Security 拦截链管理。

## 5. 数据权限模型
- **架构原则**：不在底层架构（如 MyBatis 拦截器层）做一刀切的强制数据权限拦截。
- **实现方式**：数据可见性管控按具体业务要求，在具体的 `Service` 层代码中按需拼装查询条件处理，保持架构轻量。

## 6. 权限资源定义 (Permission 实体约定)
`Permission` 表不仅是后端接口鉴权凭证，也是全量资源集合。
- **分类 (Type)**：必须通过 `type` 字段严格区分资源类型：`MENU`（菜单）、`BUTTON`（按钮）、`API`（接口）、`DATA`（数据）、`FILE`（文件）。
- **归属**：通过 `businessGroup` 字段对资源进行业务模块归类，辅助前端构建权限分配树。
